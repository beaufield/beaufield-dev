# 納品書PDF読取アプリ向け AI API 最終選定案

## 0. 文書の目的

本書は、納品書PDF読取アプリにおけるAI/API構成について、GPT・Gemini・Claudeの各提案を踏まえて、総合的に判断した最終方針を整理したものです。

対象業務は、会社に届く納品書をスキャンPDF化し、AIで明細情報を読み取り、既存の商品マスターCSVと照合してCSV出力するアプリです。

---

## 1. 前提条件

| 項目 | 内容 |
|---|---|
| 対象業務 | 納品書PDFの読み取り、明細抽出、商品マスターCSV照合、CSV出力 |
| 読み取り対象 | A4・A5のスキャンPDF |
| 処理量 | 日20枚程度、月400枚程度 |
| 仕入先 | 複数社、最大10社程度を想定 |
| 仕入先ごとのフォーマット | 異なる |
| 既存実装 | CSV出力部分は完成済み |
| 現在のPDF読取 | Claude Codeのスキルで実行中 |
| 商品マスター照合 | 既存アプリ側で実施 |
| マスター不一致時 | コードを「1」、商品名は納品書上の名称を使用 |

---

## 2. 重要な設計方針

### 2.1 AIに任せる範囲

AIに任せるのは、PDFまたはOCR結果から以下を抽出し、JSON化する部分に限定します。

| 抽出項目 | 内容 |
|---|---|
| 仕入先名 | 納品書上の仕入先名 |
| 納品日 | 納品書上の日付 |
| 納品書番号 | 存在する場合のみ |
| 商品コード | 納品書上に記載されたコード |
| 商品名 | 納品書上の商品名 |
| 数量 | 明細行ごとの数量 |
| 単価 | 明細行ごとの単価 |
| 金額 | 明細行ごとの金額 |

### 2.2 AIに任せない範囲

以下は必ず既存アプリ側で処理します。

```text
1. 商品マスターCSVとの照合
2. 商品コードの確定
3. マスター不一致時のコード「1」変換
4. 商品名の採用ルール
5. CSV出力
```

### 2.3 理由

AIに商品マスター照合やCSV出力まで任せると、以下のリスクがあります。

| リスク | 内容 |
|---|---|
| 誤照合 | 似た商品名や近いコードをAIが推測する可能性がある |
| 再現性低下 | 同じ入力でもAIの出力が揺れる可能性がある |
| 原因切り分け困難 | 読み取りミス、照合ミス、CSV出力ミスの切り分けが難しい |
| コスト増 | 商品マスターCSV全文を毎回AIに渡すと入力トークンが増える |

したがって、AIの役割は「読み取りとJSON化」に限定し、照合とCSV出力は既存アプリ側で厳密に処理します。

## 3. 各AI提案の要点

### 3.1 GPT案の要点

| 案 | 内容 | 位置づけ |
|---|---|---|
| A | Google Document AI OCR ＋ Gemini 3 Flash | 長期運用の本命 |
| B | Claude Sonnet 4.6 単体 | 短期導入候補 |
| C | Google Document AI OCR ＋ Claude Sonnet 4.6 | 精度優先候補 |

GPT案のポイントは、OCR・AI抽出・商品マスター照合・CSV出力を分離する設計です。  
これは保守性と原因切り分けの観点で妥当です。

### 3.2 Gemini案の要点

| 案 | 内容 | 位置づけ |
|---|---|---|
| 案A | Claude Sonnet系 | 精度重視 |
| 案B | Gemini Flash系 | コスト重視 |
| 案C | Google Document AI ＋ Gemini | 堅牢性重視 |

Gemini案のポイントは、Gemini Flashで十分な精度が出るならコスト面から採用し、誤読が目立つ場合はClaudeまたはOCR併用案へ進むという考え方です。

### 3.3 Claude案の要点

| 案 | 内容 | 位置づけ |
|---|---|---|
| A | Claude Sonnet 4.6 | 精度重視 |
| B | Claude Haiku 4.5 | コスト・実装バランス |
| C | GPT-4.1 mini | 最安候補 |

Claude案のポイントは、現状Claude Codeで同等処理を実装済みであり、精度・ロジックが検証済みである点を重く見ることです。

ただし、最安モデルをそのまま本番採用するのは慎重に判断すべきです。

---

## 4. 総合判断

### 4.1 最終的な本命案

最終的な本命案は以下です。

```text
Google Document AI OCR
＋
Claude Sonnet 4.6
＋
既存アプリ側で商品マスター照合・CSV出力
```

### 4.2 理由

| 理由 | 内容 |
|---|---|
| OCR分離 | スキャンPDFは専用OCRで文字情報を取得した方が安定しやすい |
| 原因切り分け | OCRミスとAI抽出ミスを分離して検証できる |
| Claude資産活用 | 現在Claude Codeで動いている実績・プロンプト資産を活かせる可能性がある |
| 精度重視 | 月400枚程度では価格差より読み取りミス削減の方が重要 |
| 保守性 | OCR、AI抽出、照合、CSV出力を分離できる |

## 5. 推奨アーキテクチャ

```text
紙の納品書
↓
スキャンPDF化
↓
Google Document AI OCR
↓
OCRテキスト・レイアウト情報を取得
↓
Claude Sonnet 4.6で明細JSON化
↓

  マスターの商品コード・商品名を採用
↓
照合失敗：
  コード「1」
  商品名は納品書上の名称をそのまま採用
↓
CSV出力
↓
要確認データのみ人が確認
```

### 6.1 対抗案：Document AI OCR ＋ Gemini 3 Flash

```text
Google Document AI OCR
```

### 6.2 この案の強み

| 観点 | 内容 |
|---|---|
| 価格 | 最も安価になりやすい |
| セキュリティ管理 | Google Cloud内に寄せやすい |
| 保守性 | OCR・AI抽出・照合を分離できる |
| 将来性 | Google Workspace / GAS / Google Cloudとの相性がよい |

### 6.3 採用条件

以下を満たす場合は、ClaudeよりGemini案を採用してよいです。

```text

2. 数量ミス 0件
3. 推測出力 0件
4. JSON不正 0件
5. 要確認率 5%以下
6. Claude案との差が小さい
```

## 8. 採用しない方がよい案

### 8.1 AIに商品マスター照合まで任せる案

以下の構成は不採用とします。

```text
PDF
＋
商品マスターCSV全文
↓
AIに照合とCSV出力まで任せる
```

### 理由

| 理由 | 内容 |
|---|---|
| 誤照合リスク | AIが似た商品名や近いコードを推測する可能性がある |
| コスト増 | 商品マスターCSV全文を毎回渡すと入力トークンが増える |
| 再現性 | 出力が揺れる可能性がある |
| 保守性 | エラー原因の切り分けが難しい |

### 8.2 価格だけで最安モデルを即採用する案

Haiku、GPT mini、Gemini Flash-Liteなどの低価格モデルは検証対象にはしてよいです。  
ただし、初回本番の第一候補にはしません。

理由は、納品書処理で最も危険なのが以下だからです。

```text
間違っているのに、正常データとしてCSV出力されること
```

価格差は月数百円〜千円程度であり、人間の確認・修正工数や仕入データ誤登録リスクに比べて小さいです。

## 10. 検証データ

### 10.1 必要枚数

最低60枚、可能なら100枚で検証します。

| 分類 | 枚数 |
|---|---:|
| きれいなスキャン | 20枚 |
| 普通のスキャン | 20枚 |
| 読みにくいスキャン | 20枚 |
| 合計 | 60枚 |

### 10.2 仕入先ごとの偏り防止

| 区分 | 枚数 |
|---|---:|
| 主要5社 | 各8〜10枚 |
| その他仕入先 | 各2〜5枚 |

### 10.3 10枚検証では不十分な理由

10枚検証は初期確認には有効ですが、月400枚運用の最終判断には少なすぎます。  
少なくとも60枚、できれば100枚で確認します。

## 11. AI出力JSON仕様

全方式で、同じJSON仕様を使用します。

```json
{
  "supplier_name": "",
  "document_date": "",
  "document_number": "",
  "rows": [
    {
      "line_no": 1,
      "raw_code": "",
      "raw_name": "",
      "quantity": "",
      "unit_price": "",
      "amount": "",
      "confidence": "",
      "notes": ""
    }
  ],
  "warnings": []
}
```

---

## 12. 共通プロンプト方針

全方式で以下を明示します。

```text
出力は必ずJSONのみ。
Markdown、説明文、コードブロック、補足文は出力しない。
読めない項目は空文字にする。
商品コードを推測で補完しない。
数量・単価・金額を推測しない。
小計、税額、送料、備考、タイトル行を商品明細として出力しない。
判定に迷う場合は warnings に理由を入れる。
```

## 14. 最終判断ロジック

### Step 1：Document AI OCR ＋ Gemini 3 Flashを検証

```text
Document AI OCR ＋ Gemini 3 Flash が合格条件を満たす
→ Gemini案を採用
```

この場合、価格・Google Cloud内での管理・保守性の面で最も有利です。

### Step 2：Gemini案で不安があれば Document AI OCR ＋ Claude Sonnet 4.6を検証

```text
Gemini案で明細行抜け・数量ミス・JSON不正・推測出力が出る
→ Document AI OCR ＋ Claude Sonnet 4.6 を採用候補にする
```

### Step 3：Claude単体は短期導入候補

```text
Claude単体が圧倒的に早く実装できる
かつ
精度がDocument AI併用案と同等以上
→ Claude Sonnet 4.6単体を短期採用
```

ただし、長期的にはOCR分離型に移行できる設計にしておきます。

### Step 4：全方式で重大エラーが出る場合

```text
全方式で明細行抜け・数量ミス・推測出力が出る
→ 完全自動化せず、確認画面必須の半自動運用にする
```

## 15. 最終結論

### 15.1 採用方針

```text
本命：
Google Document AI OCR ＋ Claude Sonnet 4.6

対抗：
Google Document AI OCR ＋ Gemini 3 Flash

短期導入：
Claude Sonnet 4.6 単体
```

### 15.2 実務上の進め方

```text
1. まずDocument AI OCRを前段に置く前提で検証する
2. AI抽出モデルとしてGemini 3 FlashとClaude Sonnet 4.6を比較する
3. Geminiで重大エラーが0件なら、Geminiを採用
4. Geminiで明細行抜け・数量ミス・推測出力・JSON不正が出るなら、Claudeを採用
5. 既存Claude Code処理の移植が非常に簡単なら、Claude単体で短期稼働も可
6. ただし本番運用では、商品マスター照合とCSV出力は必ず既存アプリ側で行う
```

### 15.3 最終判断

本番の最適案は、以下です。

```text
Google Document AI OCR ＋ Claude Sonnet 4.6
```

ただし、検証で以下を満たす場合は、Gemini案に切り替えてよいです。

```text
Document AI OCR ＋ Gemini 3 Flash が
・明細行抜け 0件
・数量ミス 0件
・推測出力 0件
・JSON不正 0件
・要確認率 5%以下
を満たす場合
```

その場合、価格・セキュリティ管理・長期保守性の面でGemini案が有利になります。

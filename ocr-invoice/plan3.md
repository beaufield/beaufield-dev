# 納品書OCRアプリ向け AI API 検証計画書（統合版）

## メタ情報

- **作成者:** 株式会社ビューフィールド
- **作成日:** 2026年4月26日
- **バージョン:** v2.0（GPT・Gemini・Claudeの3者提案を統合）
- **目的:** 納品書PDF読み取りに使うAI APIを、検証データに基づいて確定する
- **対象読者:** 社内決裁者、実装担当者、レビューを行うAI

---

## 1. システム概要

### 1.1 業務フロー

```
納品書（紙）
  → スキャン → PDF
  → AIがPDFを読み取り、明細情報をJSON出力
  → アプリ側でJSONと商品マスターCSVを照合
  → 照合結果をCSV出力（このアプリは既に完成済み）
```

### 1.2 処理ルール

- 商品マスターCSV内のコードと一致した場合：マスターのコード・商品名を使用
- 一致しない場合：コードは `"1"` を使用、商品名は納品書記載の名称をそのまま使用

### 1.3 AIの役割範囲

**AIに任せること：**
- PDFを読み取り、明細情報をJSON形式で構造化出力する

**AIに任せないこと：**
- 商品マスターとの照合
- コード変換・CSV出力
- 推測による補完（読めない値を勝手に作らない）

## 2. 要件と評価観点

### 2.1 要件

| 項目 | 内容 |
|---|---|
| 処理量 | 1日20枚、月400枚（A4・A5スキャンPDF） |
| 仕入先数 | 最大10社（フォーマットが各社異なる） |
| 呼び出し元 | GAS（Google Apps Script） |
| 現状資産 | Claude Codeのスキルで同等処理を実装・検証済み |

### 2.2 評価観点と優先順位

```
セキュリティ ＞ 読み取り精度 ＞ 価格 ＞ 実装コスト
```

理由：情報漏洩は事業継続リスク、精度低下は人件費増、価格差は月数百〜千円規模で軽微。

## 3. セキュリティ方針

### 3.1 各社APIのデータ取り扱い（2026年4月時点）

| 項目 | Anthropic（Claude） | OpenAI（GPT） |
|---|---|---|
| APIデータの学習利用 | デフォルト：なし | デフォルト：なし |
| データ保持期間 | 最大30日（安全確認目的） | 最大30日（不正利用検出目的） |
| Zero Data Retention | 申請可能 | 申請可能 |

### 3.2 共通セキュリティルール（検証時から適用）

| 項目 | ルール |
|---|---|
| API利用 | 必ず有料（Paid）API。無料枠は学習利用リスクのため使用禁止 |
| APIキー管理 | GAS の PropertiesService で保管。フロント・GitHub には一切置かない |
| ログ管理 | PDF本文・OCR全文・取引先名・商品名をログに残さない |
| エラー通知 | LINE WORKS等への通知は「ファイル名・エラー種別・処理ID」のみ |
| 保存先 | Google Drive等の保存先はアクセス権限を最小限にする |
| 監査記録 | 処理日時・処理者・方式・成功/失敗・エラー種別のみ記録 |

### 3.3 除外モデル

- **Gemini 無料枠：** データがGoogleのモデル学習に使用される可能性があるため除外
- **Document AI + LLM の2段階構成：** 実装コストが規模に対して過大なため除外（将来月1,000枚を超えた段階で再検討）

## 5. 共通プロンプト

検証時、3モデルへ同一の共通プロンプトを投入する。モデル依存の差を純粋に比較するため。

### 5.1 システムプロンプト

```
あなたは日本の納品書からデータを抽出するアシスタントです。
以下のJSONフォーマットのみを返してください。説明文・Markdown・コードブロックは出力しないでください。

{
  "supplier_name": "仕入先名",
  "document_date": "YYYY-MM-DD",
  "document_number": "納品書番号（なければ空文字）",
  "rows": [
    {
      "line_no": 1,
      "raw_code": "記載のコード（なければ空文字）",
      "raw_name": "記載の品名",
      "quantity": 数値またはnull,
      "unit": "単位",
      "unit_price": 数値またはnull,
      "amount": 数値またはnull,
      "confidence": "high / medium / low",
      "notes": "判定に迷う場合の理由（なければ空文字）"
    }
  ],
  "warnings": []
}
```

### 5.2 必須ルール

- 出力はJSONのみ。Markdownや補足文は出力しない
- 読めない項目は空文字または null にする
- 商品コード・数量・単価・金額を**推測で補完しない**
- 小計・税額・送料・備考・タイトル行を商品明細として出力しない
- 判定に迷う場合は warnings に理由を入れる

## 6. 検証データセット

### 6.1 必要枚数：30枚

10枚では統計的に不十分、60枚以上は業務負荷が大きすぎる。30枚を中庸として設定。

### 6.2 内訳

| 区分 | 枚数 | 選び方 |
|---|---|---|
| きれいなスキャン | 10枚 | 文字鮮明・主要仕入先 |
| 普通のスキャン | 10枚 | 通常品質・複数仕入先 |
| 読みにくいスキャン | 10枚 | かすれ・斜め・印影重なり等 |
| **合計** | **30枚** | 仕入先6〜10社をカバー |

### 6.3 仕入先カバレッジ

主要5社は各5枚以上、その他仕入先は各1〜2枚を含めること。

## 7. 正解データの作成

検証前に、人間が正解CSVを作成する。

| 項目 | 内容 |
|---|---|
| pdf_file_name | 元PDFファイル名 |
| supplier_name | 仕入先名 |
| document_date | 納品日 |
| document_number | 納品書番号 |
| line_no | 明細行番号 |
| raw_code | 納品書上の商品コード |
| raw_name | 納品書上の商品名 |
| quantity | 数量 |
| unit_price | 単価 |
| amount | 金額 |

**注意：** 商品マスター照合の結果ではなく、納品書に「実際に書かれている内容」を正解とする。AI評価は読み取り精度のみを評価対象とする。

## 8. 採点基準

### 8.1 配点（100点満点／1枚あたり）

| 項目 | 配点 | 採点基準 |
|---|---|---|
| 明細行数の一致 | 25点 | 正解の明細行数と完全一致 |
| 数量の一致 | 20点 | 全明細の数量が一致 |
| 商品コード（raw_code）の一致 | 15点 | 全明細のコードが完全一致 |
| 商品名（raw_name）の一致 | 15点 | 誤字・省略なく一致 |
| 単価・金額の一致 | 10点 | 全明細で一致 |
| 仕入先名・納品日 | 10点 | 両方とも正確 |
| JSON妥当性 | 5点 | パース可能 |
| confidence/warnings の妥当性 | 0点（参考） | 不正確な箇所を自己申告できているか |
| **合計** | **100点** | |

### 8.2 エラー分類（重大度別）

GPT案のエラー分類体系を採用。

| 種別 | 内容 | 重大度 |
|---|---|---|
| **E1** | **明細行抜け**（1行まるごと出力されない） | **最重大** |
| **E2** | **数量ミス**（数量が違う） | **最重大** |
| **E8** | **推測出力**（読めないのに値を作る） | **最重大** |
| E3 | コード誤読（商品コードが違う） | 高 |
| E5 | 単価・金額ミス | 高 |
| E6 | JSON不正（パースできない） | 高 |
| E4 | 品名崩れ（欠ける・別商品名になる） | 中 |
| E7 | 余計な行（小計・税額を商品行にする） | 中 |

### 8.3 合格基準

| 判定 | 条件 |
|---|---|
| **採用可能** | 平均90点以上 かつ E1/E2/E8 が0件 |
| 条件付き採用 | 平均85点以上 かつ E1/E2/E8 が3件以下（要確認画面で停止可能なら） |
| 不採用 | 平均85点未満 または E1/E2/E8 が4件以上 |

## 9. 判定基準と重み

| 判定軸 | 重み | 根拠 |
|---|---|---|
| 精度（平均点+重大エラー数） | 50% | 重大エラー1件＝15〜30分の手修正。月400枚規模で精度差は人件費に直結 |
| セキュリティ要件の充足 | 20% | 全モデル満たす前提だが、要確認事項の対応容易性で差がつく |
| 価格（月コスト） | 15% | 年間差額は最大¥10,000程度。経営判断材料としては小さい |
| 実装コスト | 15% | C（GPT）はPDF→画像変換が必要。GASでの実装可否が論点 |

## 10. 判断フロー

```
STEP 0: Cの実装可否を事前確認
  GASでPDF→画像変換が実装可能か検証
  - 不可 → Cを除外、A・Bの2択で進む
  - 可能 → A・B・C の3択で精度検証へ

STEP 1: B（Haiku 4.5）の検証結果を確認
  - 平均90点以上 かつ 重大エラー0件
    → Bを採用（コスト×精度のベスト）
  - 上記未満 → STEP 2へ

STEP 2: A（Sonnet 4.6）とBの精度差を確認
  - 平均点で10点以上の差
    → Aを採用（精度優先）
  - 差が10点未満
    → Bを採用（コスト差が精度差を上回る）

STEP 3（並行）: C（GPT-4.1 mini）の精度確認
  - 重大エラー0件 かつ 採用済み案と同等以上の精度
    → Cに変更（最安）
  - そうでなければ
    → STEP 1〜2の結果を維持

STEP 4: いずれも採用基準を満たさない場合
  → 完全自動化を諦め、確認画面付きの半自動運用を採用
```

## 13. 推奨する開始順序

### 13.1 Phase 1：実装可否の事前確認（最優先・30分〜1時間）

CのGASからのPDF→画像変換が実装可能か、最初に確認する。不可ならCは除外、検証は2モデルで済むため工数が大幅に減る。

### 13.2 Phase 2：B（Haiku 4.5）から検証開始

最もコスパが良く、合格すれば検証終了・即採用できる。不合格でもAへ切り替えるだけで済むため最短で意思決定できる。

### 13.3 Phase 3：A・C を必要に応じて追加検証

Bが合格しなかった場合のみ、Aで再検証。Cは実装可能な場合のみ並行検証。

## 14. 想定リスクと対応

| リスク | 対応 |
|---|---|
| 全モデルが90点に届かない | 確認画面付き半自動運用に切り替え。AIは下書き作成、人間が検収 |
| 特定仕入先のフォーマットだけ精度が低い | その仕入先のみ別モデルを使う2系統構成 |
| 月途中でAPIモデルが廃止される | 前世代モデルへの即時切り替え手順を準備 |
| トークン使用量が想定を超過 | 月予算上限をAPI管理画面で設定（自動停止） |
| PDFスキャン品質が向上しない | スキャナー設定の見直し（300dpi以上推奨） |
